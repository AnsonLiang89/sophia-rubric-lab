# Sophia Rubric Lab 评测方案 v3.2 提案

日期：2026-04-26

## 一、先给结论：这次应该怎么改

我同意你的判断：当前新版虽然已经朝“诊断优先”迈了一步，但还没有彻底完成切换，主要问题不是标准完全错了，而是**标准、执行、数据结构、前端呈现四层没有真正收拢到同一套阅读路径上**。

我建议下一版明确升级为：

> **v3.2 = 事实/逻辑错误优先 + 有依据的非共识洞察 + 正文主路径唯一化 + 评分总表之外全部退回正文。**

换句话说，下一版不再追求“模块很多、看起来很完整”，而是追求：

1. **先查错，再评分**；
2. **先做诊断，再做展示**；
3. **正文是主战场，结构化字段只是辅助**；
4. **每个评测观点都必须带理由、论据、必要时带原文引用**；
5. **除评分总表外，其他反馈、核验、对照、优缺点，都融入评测报告正文。**

## 二、我对当前版本的诊断

### 1. 方向已经部分对了，但还不够彻底

从现有文档看，`RUBRIC_STANDARD.md` 已经进入 v3.1，里面其实已经吸收了你刚才提出的几个核心想法：

- 事实错误 / 逻辑错误优先；
- 非共识观点必须有依据；
- 正文改成总-分-总主路径；
- 除评分总表外，其他信息尽量并入正文。

但真正的问题在于：**文档已经往前走了，界面和契约还停在上一阶段的思路里。**

### 2. 当前真正的矛盾点有四个

#### 矛盾 A：阅读主路径仍然被模块切碎了

当前报告页除了“评分总表”之外，还有：

- Sophia 聚焦诊断面板
- 每份报告的反馈
- 核验地图
- 维度 checklist 完成度
- 时间预算报表
- 最后才是评测报告正文

这会造成一个直接后果：

> **读者还没进入正文，就已经被迫在多个模块之间跳转。**

结果就是“形式感”压过了“诊断感”。你想重点读的是正文，但当前页面把正文放到了阅读路径的后半段。

#### 矛盾 B：事实核验虽然被强调了，但“错误优先级”还不够绝对

现有标准已经强调 R1 是生命线，但执行上仍然容易出现一个问题：

- 先看总分/维度分；
- 再回头补核验说明；
- 于是“重大事实错误”和“普通短板”在呈现顺序上混在一起。

这不利于诊断。

我认为下一版要把这个原则写死：

> **凡是发现事实性错误或逻辑性错误，必须优先外部查验，并且在正文中优先展开；其他维度的讨论必须排在其后。**

#### 矛盾 C：对“新意/非共识”的要求已经提出，但评价口径还不够清楚

现在的说法是“非共识但要有依据”，这是对的；但还缺一个更可执行的判定标准：

到底什么算“新意”？

下一版需要把它说得更具体：

- 不是“说了别人没说的话”就算新意；
- 不是“语气自信、结论反直觉”就算新意；
- 只有当一个观点同时满足“**非共识 + 有证据 + 有推理链 + 对决策有增量**”时，才算有效洞察。

#### 矛盾 D：版本语义已经出现漂移

当前 `.evaluations` 文档已经写到 v3.1，但前端类型和展示逻辑仍然主要按 v3.0 组织，阅读路径也还是多模块思路。这说明：

> **标准更新了，但产品化实现还没完全跟上。**

所以我建议下一步不要继续做零碎修补，而是直接提出一版完整的新方案，把“评测流程、评测标准、正文结构、UI 阅读路径、结构化字段角色”一次性对齐。

## 三、v3.2 的总原则

下一版我建议只保留四条总原则，其他设计都围绕它们展开。

### 原则 1：错误优先于优点

评测首先是诊断，不是颁奖。

所以优先级固定为：

1. **事实性错误**
2. **逻辑性错误**
3. **是否真正回答问题**
4. **是否有有依据的新意**
5. **是否足够完整、对决策有价值**

也就是说：

- 一份报告哪怕结构很漂亮，只要核心事实错了，先判错；
- 哪怕观点很多，只要推理链站不住，先判错；
- 只有在“基本可信”成立以后，再讨论它有没有洞察和启发。

### 原则 2：正文优先于模块

“评分总表”保留为唯一的独立总览模块。

除此之外：

- 每份报告反馈
- 核验地图
- 跨产品对照
- 重点错误拆解
- Sophia/竞品的优缺点与建议

**全部进入正文。**

结构化字段仍然可以保留，用于：

- 机器校验
- 聚合分析
- 调试/开发视图

但它们**不应该继续占据主阅读区**。

### 原则 3：每一条评测观点都必须可追溯

下一版的正文观点应该统一遵守一个最小单元：

> **判断 -> 理由 -> 论据 -> 影响**

具体来说，每一条正式评语都应尽量包含四件事：

1. **判断**：我认为它好/不好，或者存在什么问题；
2. **理由**：为什么这样判断；
3. **论据**：原文引用、外部来源、逻辑重建、对照案例；
4. **影响**：这会怎样影响分数、结论或用户决策。

如果一条评语只剩“结论”，没有理由和证据，那它不应该进入最终报告。

### 原则 4：新意必须以证据为边界

下一版明确鼓励：

- 有新意的建议；
- 非共识但成立的观点；
- 反直觉但有根据的判断；
- 对其他报告没有提出、但确实关键的角度。

但要同时明确：

> **无依据的新奇，不算洞察；只算幻觉或表演。**

## 四、v3.2 评测流程：从“打分流程”改成“诊断流程”

我建议把执行流程改成下面这 6 步。

## 0. 预判阶段：先识别高风险区，不急着给分

先快速读完整体报告，标记两类高风险内容：

- **承重事实断言**：数字、时间、主体、因果、政策、引用；
- **承重逻辑断言**：结论成立依赖的关键推理链。

目的不是马上做 checklist，而是先确定：

> 这份报告最可能把用户带偏的地方在哪里？

## 1. 错误优先核验：先查事实与逻辑

对所有高风险断言，按优先级核验：

### 1.1 事实核验

至少覆盖：

- 人、事、时间、地点、主体是否准确；
- 数字、比例、金额、日期、量级是否准确；
- 引用是否真实存在、是否对应原文；
- 语境是否错位。

### 1.2 逻辑核验

至少覆盖：

- 因果链是否倒置；
- 推理有没有跳步；
- 同一篇内部是否自相矛盾；
- 量纲、口径、算术是否自洽。

### 1.3 结论性要求

一旦发现事实性错误或逻辑性错误：

- 必须做外部核验或内部重建；
- 必须在正文里优先展开；
- 必须说明“错在哪里”“怎么确认它错了”“为什么这会影响判断”。

## 2. 五大维度评分：建立在核验之后，而不是之前

维度评分继续保留 R1~R5 + 额外维度，但评分顺序改成：

1. **R1 准确性**：先判；
2. **R2 相关性**：在“没跑题、没绕题”的基础上判；
3. **R3 论证深度**：重点看有没有“有依据的新意”；
4. **R4 完备性**：看关键维度有没有缺；
5. **R5 决策价值**：看有没有真正帮用户推进决策。

也就是说，**R3 的“洞察”是建立在 R1 可信的前提上**，不能让“漂亮洞察”掩盖“事实不可靠”。

## 3. 额外维度处理：从“装饰项”变成“特定 query 的放大镜”

额外维度保留，但使用原则要更严格：

- 只有当某个 query 确实存在一个**超出五大维度、但又显著影响判断**的关键差异时，才激活额外维度；
- 额外维度一旦激活，正文必须单独展开；
- 如果不激活，也可以在“额外重点问题”里讨论，但不计分。

换句话说：

> 额外维度不是为了让结构更花，而是为了捕捉本题真正的特异点。

## 4. 形成诊断结论：围绕 Sophia，但不牺牲横评价值

建议继续保留“聚焦 Sophia”的主线，但表达方式做收敛：

- **正文主线**围绕 Sophia 的问题与优化方向展开；
- **横向对标**用来解释：Sophia 到底输在哪、赢在哪、哪些是行业共性短板；
- 不再把这件事独立做成大块面板，而是写进正文相关段落里。

## 5. 输出评测报告正文：总-分-总，正文即诊断主战场

正文固定采用下面四段：

1. **评测结论**
2. **按维度展开评测结论、详情与论据**
3. **额外需要拿出来讨论或强调的问题**
4. **Sophia 及其他被评测主体做得好/不好之处，以及建议**

这四段其实就是“总-分-总”的落地版：

- 第一段是总；
- 第二、三段是分；
- 第四段是回收和总结。

## 6. 最后才是结构化落盘

`summary` 仍然保留，但定位改成：

- 给系统校验；
- 给后续聚合；
- 给次级分析；
- 不直接决定主阅读路径。

## 五、v3.2 的评分标准：五大维度怎么重新强调

这里不是推翻 R1~R5，而是重新定义每个维度的“阅读重点”和“写作要求”。

## R1 准确性：明确成为第一优先级

### 重点

R1 不只是“查几个数字”，而是要把两类问题拆开写清楚：

- **事实错误**：事实本身错；
- **逻辑错误**：事实可能没错，但推理链错了。

### 正文写法要求

R1 段必须优先回答四件事：

1. 哪些承重断言被核验；
2. 哪些是正确的；
3. 哪些是错误的；
4. 这些错误会怎样影响结论或用户决策。

### 建议新增的硬约束

- 若正文指出事实/逻辑错误，必须给出：
  - 原文引用；
  - 核验依据；
  - 影响说明。
- 若是重大错误，必须优先出现在“评测结论”或“额外重点问题”里。

## R2 相关性：不是“有没有提到”，而是“有没有正面回答”

### 重点

下一版要更清楚地区分：

- **提到了相关话题**；
- **真正回答了用户的问题**。

### 正文写法要求

R2 段至少要说明：

- 用户真正要解决的核心问题是什么；
- 这份报告有没有正面回答；
- 它是只答了字面，还是答到了潜在诉求；
- 有没有明显跑题、绕题、拿信息堆砌代替回答。

## R3 论证深度：重点考察“有依据的新意”

这是你这次特别强调的重点，我建议把它写得更硬。

### 我建议把“有效新意”定义为四选一命中

一条观点要想在 R3 里被当成高价值洞察，至少命中以下之一：

1. **提出了非共识但成立的判断**；
2. **提出了别人没有明确提出、但事实支持的关键约束**；
3. **提供了新的分析框架，而不只是换一种说法重述共识**；
4. **做到了反方视角/反例检验，从而让结论更稳。**

### 但同时必须满足两个门槛

- 有证据；
- 有推理链。

### 建议新增的硬要求

R3 的高档（S/A）至少要满足以下其一：

- 有 1 条以上“非共识但有依据”的洞察；
- 或有 1 条以上“对决策方向产生实质影响”的新框架。

如果全文只有共识性重述，即便写得很顺，也不应拿到高档。

## R4 完备性：不再奖励“表面很全”

### 重点

完备性不能再被“章节多、表格多、框架多”轻易骗过。

真正要看的不是“看起来覆盖了很多”，而是：

- 决策所需关键维度是否真的覆盖；
- 有没有关键缺口；
- 有没有假覆盖（提到但没展开）；
- 结构是否服务于论证，而不是服务于排版。

### 正文写法要求

R4 段要明确指出：

- 哪些关键维度覆盖到了；
- 哪些关键维度缺了；
- 哪些虽然提了，但实际上没有形成有效分析。

## R5 决策价值：看能不能推进下一步行动

### 重点

R5 不只看“有没有建议”，而是看：

- 建议是否具体；
- 能否执行；
- 有没有风险提示；
- 是否真的让用户更接近下一步决策。

### 正文写法要求

R5 段最好明确回答：

- 这份报告让用户新增知道了什么；
- 它给出的建议能不能真的拿来用；
- 它有没有过度自信；
- 它有没有清楚交代不确定性。

## 六、v3.2 的正文结构：正文不是摘要，而是诊断展开

我建议正文固定成下面这个骨架。

## 第一部分：评测结论

这一部分只做三件事：

1. 给出总排名、总分、是否触发 veto；
2. 先说 Sophia 本轮最核心的问题和最值得肯定的点；
3. 用 3-6 句把本轮的主诊断说透。

### 这一段必须回答的问题

- 本轮谁最好、谁最差，为什么；
- Sophia 最大的问题是什么；
- Sophia 如果只改一件事，应该先改什么。

### 这一段里保留唯一独立模块

- **评分总表**

评分总表建议就放在这一部分里，不再单独漂浮成一个“与正文平行”的阅读系统。

## 第二部分：按维度展开评测结论、详情与论据

这是正文主体，必须覆盖：

- R1
- R2
- R3
- R4
- R5
- 若有激活的额外维度，也要单列

### 每个维度的小节统一采用这个写法

1. **结论句**：这一维谁强谁弱，核心差异是什么；
2. **原因展开**：为什么这样判断；
3. **证据展开**：原文引用 / 外部核验 / 对照样本；
4. **分数归因**：为什么是这个档位，而不是更高或更低。

### 这里特别强调两点

- R1 小节必须优先展开事实/逻辑错误；
- R3 小节必须重点展开“是否有依据的新意”。

## 第三部分：额外需要拿出来讨论或强调的问题

这部分是“超出维度分数，但必须说透”的内容。

适合放这里的东西包括：

- 重大事实错误的完整拆解；
- 关键逻辑链为什么站不住；
- 某个被评分机制压平、但实际上非常重要的问题；
- 某个跨维度的问题（例如“结构上像很强，其实是无源数字堆砌出来的伪专业感”）；
- 某个行业共性短板。

### 这部分相当于把过去的这些模块收回正文

- 核验地图 -> 变成“事实/逻辑核验记录”小节
- 聚焦诊断 -> 变成“Sophia 与对手差距的根因”小节
- 每份反馈 -> 变成正文中对各主体问题的展开前置

## 第四部分：Sophia 及各个其他被评测主体做得好的和不好的地方，以及建议

这一部分是回收总结，也最接近你要的“好/不好/建议”。

### 建议固定写法

对每个主体分别写：

- **做得好**：1-3 条，指向具体维度和具体证据；
- **做得不好**：1-3 条，指向具体维度和具体证据；
- **建议**：1-3 条，必须可执行。

### 重点要求

- Sophia 的部分可以更详细；
- 其他产品可以略短，但不能空泛；
- 不能只写“见 summary”，必须在正文里展开。

## 七、呈现方式：页面只保留一条阅读主线

这里是最关键的产品层改法。

## 页面主结构

我建议报告页改成：

1. **评分总表**
2. **评测报告正文**

结束。

也就是说，读者打开一份报告后，主阅读路径只有两步：

- 先看总表；
- 再读正文。

## 其他模块怎么处理

### 1. 每份报告的反馈

不再做独立卡片区。

处理方式：

- 写进正文第四部分“各主体做得好/不好/建议”；
- 如果保留结构化字段，也只用于数据层，不再默认展示成独立模块。

### 2. 核验地图

不再做独立大面板。

处理方式：

- 作为 R1 小节和“额外重点问题”里的子表/子清单出现；
- 核验不是给人“浏览”的，是给人“理解某个错误如何被确认”的。

### 3. checklist 完成度

不再占主阅读区。

处理方式：

- 保留在 summary 里用于自检 / lint；
- 默认不展示给主阅读用户；
- 最多放进开发态折叠区，而不是正式阅读路径。

### 4. 时间预算报表

同上，不应进入主阅读路径。

它是流程质量控制信息，不是评测内容本身。

### 5. Sophia 聚焦诊断

不再单独作为一个与正文平行的面板。

处理方式：

- 其诊断内容写进“评测结论”和“各主体优缺点与建议”；
- 若仍保留结构化字段，则只为跨任务聚合服务，不在单份报告页面中抢正文位置。

## 页面设计原则一句话概括

> **评分总表负责导航，正文负责诊断。**

## 八、结构化字段和契约怎么改

我不建议把结构化字段全删掉，因为它们仍然有系统价值；但建议改变它们的角色。

## 1. 应保留的字段

建议保留：

- `overallScores`
- `rubric`
- `extraDimensions`
- `sbs`
- `claimInventory`
- `claimChecks`
- `dimensionChecklists`
- `verificationBudget`
- `perReportFeedback`
- `crossProductInsights`

## 2. 但要明确：这些字段不是主阅读界面

它们的定位改为：

- 机器校验；
- 数据聚合；
- 运维/调试；
- 未来的跨任务分析；
- 为正文生成提供底稿。

## 3. contractVersion 建议直接升到 3.2

原因不是字段大改，而是语义大改：

- 阅读主路径变了；
- 正文要求变了；
- 模块角色变了；
- “有依据的新意”从软要求升级为明确标准；
- 文档与实现需要重新对齐。

## 4. 建议新增的正文校验规则

下一版 lint / 自检可新增以下规则：

### 规则 A：正文必须是四段锚点

必须包含：

1. 评测结论
2. 按维度展开
3. 额外重点问题
4. 各主体优缺点与建议

### 规则 B：R1 段和“额外重点问题”段至少各有一处原文引用

因为这两段是最容易流于抽象判断的地方。

### 规则 C：正文中每个低分结论都必须说明原因和证据

不能只有“某维度较弱”这种判断句。

### 规则 D：R3 高档必须体现“有依据的新意”

如果给到 A/S，却通篇没有非共识但有依据的洞察，应视为口径漂移。

### 规则 E：若正文指出事实/逻辑错误，必须同时出现

- 原文引用；
- 核验依据；
- 错误影响。

## 十一、最终版方案摘要（可直接作为决策结论）

### 我建议你采纳的新版方案

1. **评测重点**：把事实性错误、逻辑性错误放在绝对第一优先级，发现后必须主动外部查验。
2. **洞察标准**：鼓励新意和非共识，但只有“有证据、有推理、对决策有增量”的新意才算有效洞察。
3. **正文要求**：每个评测观点都要有判断、理由、论据、影响，必要时引用原文。
4. **报告结构**：固定为
   - 评测结论
   - 按维度展开评测结论与论据
   - 额外重点问题
   - 各主体优缺点与建议
5. **呈现方式**：保留“评分总表”作为唯一独立模块，其他反馈、核验、诊断、对照信息全部回归正文。
6. **结构化字段角色**：保留机器可读字段，但不再让它们主导单份报告的阅读路径。
7. **版本策略**：直接升级为 v3.2，把文档、契约、前端阅读路径一次性对齐。

## 十二、我的判断

如果这次真的想把 Rubric Lab 从“评分系统”推进到“诊断系统”，那就应该做这次切换。

因为你现在真正要的，不是“一个看起来很完整的评测页面”，而是：

> **一份读完之后，能明确知道 Sophia 错在哪、别人好在哪、下一步该改什么的评测报告。**

这件事的中心一定是正文，而不是模块。
