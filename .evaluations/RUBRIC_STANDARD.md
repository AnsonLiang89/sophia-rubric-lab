## 一、评测宗旨

我们做的是**「AI 产品对 AI 产品」的横向评测**。

针对同一个 Query（商业、投研、技术、生活等真实问题），让多个 AI 产品（SophiaAI、MiroThink、ChatGPT、DeepSeek、Claude 等）各自生成一份报告，然后由评测官 Sophia 打分 + 写评测报告 + 给出 SBS 结论。

**我们要回答的核心问题是：**

> **这份报告，能不能真的帮到一个要做决策的人？**

这意味着我们衡量的是：

- 说的东西**对不对**（事实、数字、信源）
- 答的是不是**用户真正问的问题**
- 有没有**真正的分析和推导**，而不只是信息罗列
- 是否覆盖**决策所需的关键维度**
- 看完之后用户是否**多知道了什么有用的东西**、能否**帮到下一步决策**

**我们不衡量的：**

- 篇幅长度
- 排版漂亮程度（markdown 花哨度不计分）
- 语气是否讨好
- 术语堆砌感

## 二、五大维度（Rubric v2.2）

这 5 个维度是评测的骨架，**每一份评测都必须覆盖全部 5 项**，权重固定。

| ID | 名称 | 权重 | 一句话定义 |
|----|------|------|---------------|
| **R1** | **准确性**（Accuracy） | **0.40** | 事实、数据、口径、信源、内部逻辑是否都经得起核查——**生命线**；v2.2 拆分 R1a 事实(0.28) + R1b 逻辑(0.12) |
| **R2** | **相关性**（Relevance） | **0.15** | 是否正面回应用户 query、是否围绕用户真实诉求展开 |
| **R3** | **论证深度**（Reasoning） | **0.20** | 观点有证据、推导闭环、分析有深度、能呈现非共识视角 |
| **R4** | **完备性**（Completeness） | **0.10** | 决策所需的关键维度是否齐全、结构是否支撑论证 |
| **R5** | **决策价值**（Decision Value） | **0.15** | 是否真正帮到用户、有无信息熵增量、有无启发、有无可操作 take-away |

权重加总 = 1.00。

> **打分通用流程（v2.2 新增硬约束，所有维度适用）**：
>
> 1. 先机械过完该维度的「必查 checklist」—— 每项写成结构化结论（是/否/部分），记入 `summary.dimensionChecklists`
> 2. 再对照「双轴 tier 判定表」定档 —— 先判两个轴的位置（强/中/弱），交叉查表得 tier（S/A/B/C/D）
> 3. tier 定了，score 就机械映射（10/8/6/4/2），**禁止**反向凑分、禁止定性词自由打分
> 4. 如有历史样本锚点（同维度 S/A/B/C/D 的已发产物链接）就先对齐口径，再下档位

### R1 · 准确性（Accuracy）· 权重 0.40

**核心问：这份报告说的东西，对不对？支撑它说的证据，硬不硬？内部推导链条站不站得住？**

准确性是评测的**生命线**。v2.2 在 v2.1 基础上做两件事：
- **事实层 vs 逻辑层显式拆分**：R1a（事实/数字/信源/口径/语境）权重 0.28；R1b（内部逻辑自洽/推导链/量纲 sanity-check）权重 0.12。两者合计仍为 0.40，对外披露时保留 R1 总分，同时在 `rubric[].subscores` 透出 R1a / R1b 分档
- **Claim 驱动的核验流程**：强制从每份报告抽取「承重 claim 清单」再逐条核验，告别"靠感觉挑几个数字对一下"

#### 判定子项（MECE）

| 子项 | 归属 | 判什么 | 怎么看 |
|---|---|---|---|
| **事实准确** | R1a | 事件、人物、因果、定性判断是否真实成立 | 逐条可核查的事实命题，交叉外部信源 |
| **数字准确** | R1a | 数值、金额、比例、日期、单位、量级是否正确 | 关键数字对照一手来源 |
| **口径统一** | R1a | 同一指标跨段落口径是否一致（营收含税/不含税、净利/扣非、YoY/QoQ） | 看口径声明 + 跨段落交叉 |
| **语境准确** | R1a | 数据 / 引用 / 事件是否放在对的时间 / 范围 / 主体上 | 看上下文匹配 |
| **信源可核查** | R1a | 关键事实和数字是否有可追溯的引用，是否一手、层级是否清晰 | 看引用规范度 + 一手源占比 |
| **内部逻辑自洽** | R1b | 报告内部各段落之间有无自相矛盾、前后不一致、算术错 | 跨段落交叉读，关键数字算一遍 |
| **量纲与因果链** | R1b | 单位/量纲对不对、"因为→所以"的链条推得通不通 | 做量纲校验、重建因果链 |

#### R1 双轴 tier 判定表

两轴：**承重 claim 核验命中率**（横轴） × **内部逻辑自洽度**（纵轴）。先判两轴位置，再交叉查表：

| 逻辑自洽度 ↓ ／ 核验命中率 → | **≥95%** | **85~95%** | **70~85%** | **50~70%** | **<50% 或触 veto** |
|---|---|---|---|---|---|
| **完全自洽，无算术错** | S (10) | A (8) | B (6) | C (4) | D (2) |
| **少量小矛盾（1~2 处边角）** | A (8) | A (8) | B (6) | C (4) | D (2) |
| **多处矛盾 / 1 处承重段自相矛盾** | B (6) | B (6) | C (4) | C (4) | D (2) |
| **承重段算术错 / 因果倒置** | C (4) | C (4) | C (4) | D (2) | D (2) |
| **编造信源 / 量级错 / 主体错位** | D (2)＋ veto | D (2)＋ veto | D (2)＋ veto | D (2)＋ veto | D (2)＋ veto |

> 核验命中率定义：`已核验承重 claim 中 status = verified-correct 的条数 / 已核验总条数`（skipped-time-budget 和 skipped-out-of-scope 不计入分母）。

#### R1 必查 checklist（7 项，全部写入 `dimensionChecklists.R1`）

1. [ ] 是否抽取了承重 claim 清单？Top 5 封顶，其中 ≥1 条属于 logic 类？
2. [ ] 承重 claim 核验覆盖率是否 ≥85%（verified + refuted 条数 / 总条数，skipped 不进分母）？
3. [ ] 每份报告是否选了 3~5 条"精确承重数字"对一手源核验？
4. [ ] 所有 veto 候选是否完成了**至少一次**外部验证后才下结论？
5. [ ] 关键量纲/占比/汇率是否都动手算过一遍？
6. [ ] 跨段落是否做了至少 1 处口径一致性交叉检查？
7. [ ] 核验过程是否都落到了 report 正文的「事实核验记录」章节？

#### R1 承重 claim 抽取规则

承重 claim = 满足任一条件的陈述：

- 这条事实/数字**如果错了，会直接影响用户决策方向**（例："XX 税率 20%"错成"5%"会让决策反转）
- 这条是报告**核心论证骨架**的锚点（承担 ≥ 1 段核心段落的论据作用）
- 这条数字**精确到小数点、亿/万、月/日**（精确的数字承担读者对信源信任）
- 这条是报告**主要结论**或 take-away 的基础

每份报告抽 Top 5（超过 5 条就按重要性排序只留前 5；不足 5 条可以少，但 ≥3 条，且必须有 ≥1 条 logic 类）。所有抽取项记入 `summary.claimInventory`，每条核验结果记入 `summary.claimChecks`。

#### R1 三阶段核验 SOP（时间盒内做）

| Pass | 目标 | 时间预算 | 做什么 |
|---|---|---|---|
| **Pass 1 快筛** | 定位嫌疑 | 10 min | 通读全部报告，对照 claimInventory 快速打标签（clean / suspicious / unverifiable-yet） |
| **Pass 2 深核嫌疑** | 落锤定性 | 7 min | 对所有 suspicious 项发起外部搜索、算术核算，确认 verified-correct / refuted / inconclusive |
| **Pass 3 逻辑一致性** | R1b 子项 | 4 min | 跨段落口径对齐、因果链重建、算术交叉——专门覆盖 R1b |

时间不够时：
- **优先保 Pass 1 全覆盖**（不能跳）
- Pass 2 按 claim 重要性排序，处理不完的标 `skipped-time-budget`，在 `verificationBudget` 和「事实核验记录」章节都注明
- Pass 3 至少要做 2 条跨段落一致性检查，不够也要标注

#### 🚨 一票否决规则（硬约束；v2.2 判定清单具体化）

**R1 若存在「重大事实错误」——满足下列 5 种模式任一——最终评级锁死在「合格」档上限，总分封顶 6.9。**

| 模式代号 | 模式名 | 判定描述（可判别） |
|---|---|---|
| **V1** | **量级错** | 关键承重数字量级错误（"600 亿美元"写成"600 万美元"；百分比小数点错位） |
| **V2** | **主体错** | 关键主体张冠李戴（把 A 公司业务说成 B 公司、把 A 国政策写成 B 国） |
| **V3** | **时间错** | 关键时间错位（引用已失效 ≥ 1 年的政策当现行；把历史人物对应到当代职位） |
| **V4** | **因果倒** | 关键因果倒置（"A 导致 B"写成"B 导致 A"、颠倒先后顺序导致论证反向） |
| **V5** | **编造源** | 引用了经外部核验**不存在**的文献/公告/人物言论（或真实存在但与报告引用内容无关） |

**触发条件（必须同时满足 3 条）**：

1. 错误出现在**承重 claim 清单**里（不在 claimInventory 的错算 R1 内扣档但不触发 veto）
2. 经外部核验**落锤**（Pass 2 命中 refuted 或发现编造）；仅凭"没给信源"不能 veto
3. 错误属于上述 5 种模式之一

触发后 payload 必填 `vetoTriggered: true` 和 `vetoReason`（引用具体 claim id + 落在 V1~V5 哪个模式）。

#### R1 参考样本锚点（随历史 outbox 积累）

| Tier | 特征简述 | 参考 evaluationId |
|---|---|---|
| **S** | 承重 claim 全命中、逻辑全自洽、一手源占比 >60% | _(待填：从历史 S 档 R1 案例中挑)_ |
| **A** | 命中率 >85%、≤2 处边角小错、无承重段矛盾 | _(待填)_ |
| **B** | 命中率 70~85% 或 1 处承重段矛盾 | _(待填)_ |
| **C** | 命中率 <70% 或承重段算术错 | _(待填)_ |
| **D** | 触发 V1~V5 任一 | _(待填)_ |

> 首轮 v2.2 评测启用时，若本表为空，评测官可以只靠双轴表定档，并把本轮产物反填为下一轮锚点。

### R2 · 相关性（Relevance）· 权重 0.15

**核心问：用户问什么，报告是不是正面答了什么？**

#### 判定子项

| 子项 | 判什么 |
|---|---|
| **正面回应** | 用户 query 的核心诉求是否被**直接回答**（不绕弯、不避重就轻） |
| **诉求覆盖** | query 里多个子问题 / 隐含诉求是否都被触达 |
| **锚点命中** | query 里的关键锚点（时间 / 地域 / 主体 / 场景）是否被正确识别并响应 |
| **潜在问题挖掘** | 能否识别 query 字面之下的**真问题/深层诉求**并主动回应 |
| **内容贴题** | 正文是否围绕 query 展开，无跑题、无大段无关填充 |

#### R2 双轴 tier 判定表

两轴：**核心诉求正面回应度**（横轴） × **锚点+潜在诉求识别度**（纵轴）

| 识别度 ↓ ／ 正面回应 → | **全部直答** | **主要直答** | **部分回避** | **答非所问** |
|---|---|---|---|---|
| **锚点全命中 + 主动挖潜在诉求** | S (10) | A (8) | B (6) | C (4) |
| **锚点全命中 + 未挖潜在** | A (8) | A (8) | B (6) | C (4) |
| **锚点部分缺失** | B (6) | B (6) | C (4) | D (2) |
| **核心锚点完全缺位** | C (4) | C (4) | D (2) | D (2) |

#### R2 必查 checklist（5 项）

1. [ ] 列出了 query 的核心诉求（≥1 条）和锚点（时间/地域/主体/场景）？
2. [ ] 每个核心诉求是否都能在报告里找到正面回应段落？
3. [ ] 每个锚点是否都被报告正确识别与响应？
4. [ ] 是否识别并回应了 query 字面下的潜在诉求（至少尝试，无则说明）？
5. [ ] 正文是否存在跑题或无关填充段（>10% 篇幅无关即扣档）？

**典型反例**（自动扣档）：

- query 问"XX 公司最新一季财报"，答的却是"XX 行业综述"
- query 要"最新时政锚点"，报告通篇无时政相关内容
- query 要"对比 A 和 B"，报告只讲了 A
- query 隐含"我在 XX 场景下做决策"，但报告通篇只做字面应答、没有任何对决策场景的回应

### R3 · 论证深度（Reasoning）· 权重 0.20

**核心问：报告不只是在"罗列"，而是在"分析"——观点有证据，推导能闭环，甚至能看到别人看不到的东西。**

#### 判定子项

| 子项 | 判什么 |
|---|---|
| **论证可支撑** | 每一个观点 / 判断是否有证据或推导支撑；论据 → 结论链条是否闭环 |
| **分析深度** | 有没有"为什么"的追问，追到二阶、三阶原因 |
| **非共识信号** | 有没有独家洞察、反直觉但成立的判断、差异化视角 |
| **多视角平衡** | 是否呈现对立视角 / 反方证据 |

> R1 与 R3 的 MECE 分工：
> - **R1a 问**："你说的这个数字/事实对不对？" → 事实本身对错
> - **R1b 问**："你这份报告内部的推导有没有自相矛盾？" → 内部一致性
> - **R3 问**："你基于这些事实得出的结论，放到更大的问题空间里，合不合理、有没有深度？" → 对外的推理链条强度

#### R3 双轴 tier 判定表

两轴：**论证闭环度**（横轴） × **分析深度 + 非共识浓度**（纵轴）

| 深度 ↓ ／ 闭环度 → | **全部闭环** | **大体闭环** | **部分跳跃** | **频繁跳跃** |
|---|---|---|---|---|
| **多处非共识 + 三阶追问** | S (10) | A (8) | B (6) | C (4) |
| **偶有非共识 + 二阶追问** | A (8) | A (8) | B (6) | C (4) |
| **只有一阶分析** | B (6) | B (6) | C (4) | D (2) |
| **几乎是信息罗列** | C (4) | C (4) | D (2) | D (2) |

#### R3 必查 checklist（5 项）

1. [ ] 随机抽 3 个核心判断，每个是否都能找到支撑证据/推导？
2. [ ] 是否有 ≥1 处超越"是什么"的二阶/三阶追问？
3. [ ] 是否出现非共识但有据的观点（反直觉判断、差异化视角）？若有，是否同时给出“观点 -> 证据 -> 推理链”？
4. [ ] 关键争议点是否呈现了对立视角或反方证据？
5. [ ] 论据 → 结论的链条是否存在明显跳跃或循环论证？

> v3.2 执行提示：发现事实错误/逻辑错误后，应先写核验结论再给维度分数；不要先给分再补证据。评分总表之外的诊断内容，都应进入正文四段主结构。

### R4 · 完备性（Completeness）· 权重 0.10

**核心问：从决策视角看，关键维度是不是都覆盖了？结构能不能支撑论证？**

#### 判定子项

| 子项 | 判什么 |
|---|---|
| **维度齐全** | 该 query 决策所需主要分析维度是否都覆盖 |
| **结构合理** | 章节安排是否支撑结论、有无重复/空白 |
| **定量深度** | 是否有必要的定量测算 / 表格 / 模型支撑（适用时） |

#### R4 双轴 tier 判定表

两轴：**决策关键维度覆盖度**（横轴） × **结构与定量支撑度**（纵轴）

| 结构/定量 ↓ ／ 维度覆盖 → | **全覆盖** | **覆盖主要** | **有明显遗漏** | **核心缺位** |
|---|---|---|---|---|
| **结构清晰 + 定量到位** | S (10) | A (8) | B (6) | C (4) |
| **结构清晰 + 定量偏弱** | A (8) | A (8) | B (6) | C (4) |
| **结构基本合理 + 无定量** | B (6) | B (6) | C (4) | D (2) |
| **结构混乱 / 章节重复空白** | C (4) | C (4) | D (2) | D (2) |

#### R4 必查 checklist（5 项）

1. [ ] 列出了"做这个决策需要覆盖的关键维度清单"（≥3 项）？
2. [ ] 每个关键维度是否都在报告里找到对应段落？
3. [ ] 章节结构是否支撑结论（无孤立段落、无重复、无明显空白）？
4. [ ] 适用定量分析的场景下，是否有必要的数据表/测算/模型？
5. [ ] 是否存在"看起来覆盖但只是一笔带过"的虚假覆盖？

### R5 · 决策价值（Decision Value）· 权重 0.15

**核心问：这份报告看完之后，用户比看之前多知道了什么有用的东西？能不能帮他做下一步决定？**

#### 判定子项

| 子项 | 判什么 |
|---|---|
| **信息增量** | 是否给出用户自己搜不到 / 想不到的信息？ |
| **启发性** | 是否提供了新角度、新框架、新联想，让用户对问题的理解**有跃迁**？ |
| **可操作 take-away** | 是否给出具体、清晰、可执行的建议或结论？ |
| **风险与不确定性诚实度** | 是否主动点出 caveat、不确定项、承认"这里我没把握"？ |

#### R5 双轴 tier 判定表

两轴：**信息增量 + 启发性**（横轴） × **可操作性 + 诚实度**（纵轴）

| 可操作+诚实 ↓ ／ 增量+启发 → | **多处增量+明显启发** | **有少量增量** | **只是信息重组** | **几乎零增量** |
|---|---|---|---|---|
| **具体可执行 + 主动标 caveat** | S (10) | A (8) | B (6) | C (4) |
| **较具体可执行 + 偶有 caveat** | A (8) | A (8) | B (6) | C (4) |
| **建议模糊 / 无 caveat** | B (6) | B (6) | C (4) | D (2) |
| **空泛废话 / 对不确定打包票** | C (4) | C (4) | D (2) | D (2) |

#### R5 必查 checklist（5 项）

1. [ ] 列出报告里的 take-away / 建议条目（≥2 条）？
2. [ ] 这些 take-away 是否具体可执行（而非"建议谨慎关注"这类废话）？
3. [ ] 报告是否给出了用户自己公开搜索难以得到的信息/观察？
4. [ ] 是否提供了让用户"换个角度想问题"的新框架/新联想？
5. [ ] 对明显不确定的事项是否主动标了 caveat？

**典型反例**（自动扣档）：

- 通篇都是维基百科级别的公开信息重述，零增量
- 只有"总体向好 / 需警惕风险"这种废话型结论，零启发
- 给出"建议谨慎关注"这种零操作性的敷衍 take-away
- 对明显不确定的信息打包票，不给 caveat

## 三、评分机制（档位打分法）

### 5 档制

每个维度**只允许**从下面 5 个档位中选一个：

| 档位 | 分数 | 含义 |
|---|---|---|
| **S** | **10** | 卓越 —— 业内最强水平，挑不出毛病 |
| **A** | **8** | 优秀 —— 明显高于平均，可直接交付 |
| **B** | **6** | 合格 —— 基本可用但有明显短板 |
| **C** | **4** | 待改进 —— 有显著缺陷 |
| **D** | **2** | 不合格 —— 结构性问题，不可用 |

**硬约束：**

- 每个维度的 score 只能是 **10 / 8 / 6 / 4 / 2** 中的一个整数
- 不允许 0.5、0.1 之类的小数
- 不允许 9、7、5、3、1 之类"档间分"
- **tier 先定、score 后映射**，禁止反向凑分

### R1 子档（v2.2 新增）

R1 由 R1a（0.28） + R1b（0.12）合成，合计 0.40。两种写法都允许：

- **合成分**：仅写 R1 一档（S/A/B/C/D + 10/8/6/4/2），R1a/R1b 放 `subscores` 字段，两个子档的加权和须与 R1 档位一致（±1 档内偏差允许，超出须在 comment 说明）
- **拆分分**：R1a 和 R1b 各自独立档位，R1 的合成分按 `R1a.score × 0.7 + R1b.score × 0.3` 机械计算（四舍五入到最近的 10/8/6/4/2 档）

> 权重比例 0.7 : 0.3（即 0.28 : 0.12）固定，不允许评测官自行调整。

### 总分计算

```
overallScore = Σ(Ri.score × Ri.weight)
```

权重 40/15/20/10/15，所以一般落点例如：

| 组合（R1/R2/R3/R4/R5） | 总分 | 评级 |
|---|---|---|
| S/S/S/S/S（10×5） | 10.0 | 🏆 卓越 |
| A/A/A/A/A（8×5） | 8.0 | ⭐ 优秀 |
| S/A/A/A/A | 8.8 | 🏆 卓越 |
| A/A/S/A/A | 8.4 | ⭐ 优秀 |
| A/A/A/B/A | 7.8 | ⭐ 优秀 |
| B/A/A/A/A | 7.2 | ⭐ 优秀 |
| B/A/S/A/S | 7.5 | ⭐ 优秀 |
| C/A/A/A/A | 6.4 | ✅ 合格 |
| 一票否决触发（R1 重大错误） | 封顶 6.9 | ✅ 合格（上限） |

**算完即定，不反向凑分**。

### 评级档位

| 总分区间 | 评级 | 含义 |
|---------|------|------|
| 8.5 ~ 10.0 | 🏆 **卓越** | 接近无可挑剔，可以直接交付决策 |
| 7.0 ~ 8.4 | ⭐ **优秀** | 质量明显高于大多数，需少量修订即可用 |
| 5.5 ~ 6.9 | ✅ **合格** | 基本可用，但存在可见短板 |
| 4.0 ~ 5.4 | ⚠️ **待改进** | 有明显缺陷，不建议直接采用 |
| 0 ~ 3.9 | ❌ **不合格** | 结构性问题，需要重写 |

> 一票否决规则生效时：无论维度分数如何，总分封顶 6.9，评级封顶"合格"。

### 辅助字段（不影响总分）

每个维度的分数旁可以附带两个辅助字段：

- **`confidence`**：`high` / `medium` / `low` —— 评测官对这个打分的确信度
- **`issueTags`**：问题标签数组（见第五节词表）

## 四、扩展维度（Extra Dimensions）

v2 起扩展维度从"可选补丁"升级为**"一等公民"**。

### 使用原则

- **每次评测至少尝试一次**——要么列出 1~3 个扩展维度，要么显式声明"本次无必要扩展"并说明原因
- 每个扩展维度必须提供 `rationale`
- 扩展维度数量上限 3 个

### 激活规则（选做，最多 1 个）

- 激活权重可选：**0.05 / 0.10 / 0.15** 三档
- R1~R5 原权重按比例等比缩减，保持总和 1.0
- 激活的 X 维度同样使用 5 档制（10/8/6/4/2）
- 未激活的 X 维度不计入总分，只作为开放观察项

**举例**：激活 X1 权重 0.10：

```
原权重：R1/R2/R3/R4/R5 = 40/15/20/10/15
缩减后：R1/R2/R3/R4/R5 = 36/13.5/18/9/13.5 + X1=10
```

### 扩展维度候选词表（参考，可自拟）

`本地化适配` · `代码正确性` · `创意密度` · `情感共鸣` · `合规敏感度` · `时效敏感度` · `交互友好度` · `技术细节颗粒度` · `数据可视化质量` · `阅读成本` · `风格契合度`

## 六、SBS 规则（Side-by-Side 对比）

**对比粒度：AI 产品对 AI 产品**（不是报告对报告；即使同一产品提交了多份报告，也先聚合到产品层再两两对比）。

**对数量：**

| 产品数量 | SBS 对数 |
|---------|---------|
| 2 个 | 1 对 |
| 3 个 | 3 对（两两组合） |
| 4 个 | 6 对 |

**每对输出（v2.2 结构升级）：**

| 字段 | 必填 | 说明 |
|---|---|---|
| `reportIdA` / `reportIdB` | ✅ | 参与对比的两份报告 id（v2.2 起明确标注 A/B 身份，避免前端歧义） |
| `winner` | ✅ | `A` / `B` / `draw` |
| `margin` | ✅ | `overwhelming`（压倒性） / `clear`（明显优势） / `slight`（略微领先） / `tie`（势均力敌，仅 winner=draw 时允许） |
| `dimensionDriver` | ✅ | 指明主要由哪个维度拉开差距（`R1` / `R2` / `R3` / `R4` / `R5` / `X1~X3`，多个用数组） |
| `keyReason` | ✅ | 一句话点出关键差异（例："A 在 R1/R3 全面领先、R5 take-away 更强；B 因 R1 V5 编造信源触发 veto 被封顶"） |

#### SBS margin 判定清单（v2.2 新增）

按**总分差**和**档位差**交叉判定：

| 总分差 Δ ／ 维度档位差 | **≥2 档（多个维度）** | **1 档（多个维度）** | **仅个别维度 1 档** |
|---|---|---|---|
| **≥1.5** | overwhelming | clear | clear |
| **0.8~1.5** | clear | clear | slight |
| **0.3~0.8** | clear | slight | slight |
| **<0.3** | slight | slight | tie (仅 draw) |

触发 veto 的一方默认输给未触发一方（margin 至少 clear）。

## 七、45 分钟时间盒 SOP（v2.2 新增）

每次评测**严格 45 分钟上限**。超时按阶段兜底，不允许无限展开。

| 阶段 | 时长 | 做什么 | 产物 |
|---|---|---|---|
| ① 读报告 | 7 min | 通读全部报告，记笔记 | 脑内地图 |
| ② Claim Inventory | 7 min | 对每份报告抽 Top 5 承重 claim（含 ≥1 条 logic） | `summary.claimInventory` |
| ③ Pass 1 快筛 | 10 min | 对所有 claim 打标签：clean / suspicious / unverifiable-yet | `claimChecks` 首版 |
| ④ Pass 2 深核 | 7 min | 外部搜索 / 算术核算 suspicious 项，落锤 verified-correct / refuted / inconclusive | `claimChecks` 终版 |
| ⑤ Pass 3 逻辑 | 4 min | 跨段落口径 + 因果链 + 算术交叉，专攻 R1b | R1b 档位 |
| ⑥ 打分 | 5 min | 过 5 份 checklist、查双轴表、定 tier、机械映射 score、算 overallScore、定 SBS | `summary.rubric` 等 |
| ⑦ 反馈+正文 | 5 min | 写 perReportFeedback + 四段结构 report 正文 | `summary.perReportFeedback` + `report` |

**超时兜底规则**：

- **任一阶段超 20% 时间**即进入下一阶段，未完成项按对应规则标 `skipped-time-budget`
- 核心阶段①~⑥ 绝对不允许跳过，即使时间剩余很少也要机械走完 checklist 和打分
- 把实际各阶段耗时和跳过的 claim 数记入 `summary.verificationBudget`
- 时间超支未必降分——评测官已经按 SOP 走完流程但部分 claim 未核验，属于"合理不完整"，只影响 R1 的 confidence

**`skipped-*` 状态定义**：

| 状态 | 用在哪 | 含义 | 对 R1 核验命中率的影响 |
|---|---|---|---|
| `skipped-time-budget` | claimChecks[].status | 时间不够，没核到 | 不计入分母（也不计入分子） |
| `skipped-out-of-scope` | claimChecks[].status | 评测官盲区 / 需要专家领域知识才能验证 | 不计入分母；R1 不扣分，但在报告里说明 |
| `inconclusive` | claimChecks[].status | 核验过但证据不足以落锤 | 计入分母（算在"未命中"一侧，拉低命中率） |
